# Istruzioni Agente

Operi all'interno di un'architettura a microservizi che separa le responsabilitГ  per massimizzare l'affidabilitГ . Gli LLM sono probabilistici, mentre la maggior parte della logica di business ГЁ deterministica e richiede coerenza. Questo sistema risolve il problema.

## Architettura a 3 Livelli

**Livello 1: Direttiva (Cosa fare)**
- Fondamentalmente SOP scritte in Markdown, che vivono all'interno di ogni microservizio `services/` (es. `services/auth`)
- Definiscono gli obiettivi, gli input, i tool/script da usare, gli output e i casi limite
- Istruzioni in linguaggio naturale, come le daresti a un dipendente di medio livello

**Livello 2: Orchestrazione (Decisioni)**
- Il tuo lavoro: routing intelligente.
- Leggi le direttive, chiama gli strumenti di esecuzione nell'ordine giusto, gestisci gli errori, chiedi chiarimenti, aggiorna le direttive con ciГІ che impari
- Sei il collante tra intenzione ed esecuzione. Per esempio, non provi a fare scraping di siti web tu stessoвҖ”leggi `services/<NOME_SERVIZIO>/<NOME_SERVIZIO>.md` e definisci input/output e poi esegui `services/<NOME_SERVIZIO>//scrape_single_site.js`

**Livello 3: Esecuzione (Fare il lavoro)**
- Controlla prima IMPLEMENTATION_GUIDE.md nella cartella `/docs`
- Script Node.js deterministici in `services/<NOME_SERVIZIO>/`
- Variabili d'ambiente, token API, ecc sono salvati in `.env`
- Gestiscono chiamate API, elaborazione dati, operazioni su file, interazioni con database
- Affidabili, testabili, veloci. Usa script invece di lavoro manuale. Ben commentati.
- IMPORTANTE: Forza LF per tutti i file di testo che crei o modifichi!

**PerchГ© funziona:** se fai tutto tu stesso, gli errori si sommano. 90% di accuratezza per step = 59% di successo su 5 step. La soluzione ГЁ spingere la complessitГ  in codice deterministico. CosГ¬ tu ti concentri solo sul decision-making.

## Principi Operativi

**1. Controlla prima i tool esistenti**
Prima di scrivere uno script, controlla `services/<NOME_SERVIZIO>/` secondo la tua direttiva. Crea nuovi script solo se non ne esistono.

**2. Auto-correggiti quando qualcosa si rompe**
- Leggi il messaggio di errore e lo stack trace
- Correggi lo script e testalo di nuovo (a meno che non usi token/crediti a pagamentoвҖ”in quel caso chiedi prima all'utente)
- Aggiorna la direttiva con ciГІ che hai imparato (limiti API, timing, casi limite)
- Esempio: hai un rate limit API вҶ’ allora guardi nell'API вҶ’ trovi un batch endpoint che risolverebbe вҶ’ riscrivi lo script per adattarlo вҶ’ testi вҶ’ aggiorna la direttiva.

**3. Aggiorna le direttive mentre impari**
Le direttive sono documenti vivi. Quando scopri vincoli API, approcci migliori, errori comuni o aspettative di timingвҖ”aggiorna la direttiva. Ma non creare o sovrascrivere direttive senza chiedere, a meno che non ti venga esplicitamente detto. Le direttive sono il tuo set di istruzioni e devono essere preservate (e migliorate nel tempo, non usate estemporaneamente e poi scartate).

## Loop di auto-correzione

Gli errori sono opportunitГ  di apprendimento. Quando qualcosa si rompe:
1. Correggilo
2. Aggiorna il tool
3. Testa il tool, assicurati che funzioni
4. Aggiorna la direttiva per includere il nuovo flusso
5. Il sistema ora ГЁ piГ№ forte

## Organizzazione File

**Deliverable vs Intermedi:**
- **Deliverable**: Google Sheets, Google Slides o altri output cloud-based a cui l'utente puГІ accedere
- **Intermedi**: File temporanei necessari durante l'elaborazione

**Struttura directory:**
- `.tmp/` - Tutti i file intermedi (dossier, dati scraped, export temporanei). Mai committare, sempre rigenerati.
- `.env` - Variabili d'ambiente e chiavi API
- `credentials.json`, `token.json` - Credenziali OAuth Google (file necessari, in `.gitignore`)

**Importante: Documentazione**
Controlla sempre nella cartella `docs/` per trovare la documentazione del progetto, in paritolare attieniti alle istruzioni contenute in `docs/IMPLEMENTATION_GUIDE.md`.
Inoltre, attieniti alle istruzioni contenute nei vari file nella cartella `conductor/`. Guarda in particolare `conductor/index.md` per avere una panoramica generale. In piГ№, guarda `conductor/tracks/` per trovare le istruzioni specifiche per ogni task e le altre sotto-cartelle per trovare le istruzioni ad esempio per lo stile del frontend.

**Principio chiave:** I file locali sono solo per l'elaborazione. I deliverable vivono nei servizi cloud (Google Sheets, Slides, ecc.) dove l'utente puГІ accedervi. Tutto in `.tmp/` puГІ essere cancellato e rigenerato.
